# 外部项目与源码参考

本页记录使用 Agent Reach、GitHub 公开 API、B站搜索和网页读取得到的类似项目线索。目的不是复制源码，而是提炼可验证的产品形态、数据字段、测试颗粒度和风险边界。

## 1. 本轮检索范围

2026-06-21 本轮只纳入可复查的公开材料：

- 公开网页和古籍页面用于资料来源核验。
- B站搜索只用于观察课程目录和案例线索，不直接升级为规则。
- GitHub 公开 API、README、目录和 raw 源码抽样用于参考产品形态、数据字段和测试颗粒度。
- 需要登录态或授权的平台暂不进入规则来源池。

GitHub 搜索只做轻量 metadata、README 和少量目录/文件抽样，不做大规模源码抓取。2026-06-21 已对 `bopo/najia` 的 README、`const.py`、`utils.py` 和测试文件做 raw 源码抽样，用于排盘回归。

## 3. 对本项目的设计结论

1. 本项目第一阶段继续保持“资料库 + 搜索 + 案例复盘”，不要变成泛命理平台。
2. 后端接口先只做只读搜索和 Schema 输出，避免还没有账户/数据库时引入伪在线案例库。
3. 装卦器已按 `bopo/najia` 的颗粒度补入卦符、卦名、变爻、卦宫、六亲、六神、世应、纳甲和伏神测试；旬空、月日和旺衰仍待下一阶段。
4. AI 解读必须和排盘/规则数据分层；AI 只能解释证据链，不能直接写入规则真值。
5. UI 保持工作台形态，保留免责声明，不做“高准确率”营销。

## 4. 准确度检测路线

准确度不能靠项目 README 宣称，需要本库自己建立回归集：

1. 古籍固定案例：从《增删卜易》抽真实卦例，记录原判断和反馈。
2. 排盘回归：同一卦输入必须稳定输出本卦、变卦、六亲、六神、世应、旬空。
3. 规则回归：每条规则至少对应一个正例和一个边界例。
4. 现代反馈案例：只收有实际反馈的案例；无反馈只算练习。
5. 每满 20 条真实案例，复盘一次规则命中、偏差和冲突。
